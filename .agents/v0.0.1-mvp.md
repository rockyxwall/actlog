# ACTLog v0.0.1 MVP Specification (Rust Desktop App)

This document defines the absolute minimum viable product (v0.0.1) for the ACTLog desktop application. The Chrome extension UI will be developed in a separate repository; this repository focuses entirely on the background collector, database, and local REST API.

---

## 1. Scope & Goals

The MVP focuses on establishing the core system service:
- Tracking the active foreground window on Windows.
- Tracking system idle (AFK) states.
- Persisting tracking data to a local SQLite database with heartbeat merging.
- Serving this data via a local synchronous REST API for the external Chrome Extension.

### In Scope for v0.0.1
1. **Single Background Binary (`actlog.exe`)**:
   - **Tray Icon**: Powered by `tray-icon` and `muda` containing a single "Quit" menu option.
   - **Win32 Window Tracker**: Polling every ~3s via `GetForegroundWindow`.
     - Handle UWP wrappers (`ApplicationFrameHost.exe`) via child window enumeration.
     - Handle elevated/system processes (`ERROR_ACCESS_DENIED`) by logging them as `"<Elevated Process>"`.
     - Skip cycles when `GetForegroundWindow` returns NULL.
   - **Basic AFK Detection**: Polling `GetLastInputInfo`. If idle time ≥ 180s, log as `source=afk, app=idle`.
   - **Database & Heartbeat Merge**:
     - SQLite file `actlog.sqlite` created next to the executable.
     - Single-writer connection for tracker thread; separate reader connection for HTTP server thread. WAL mode enabled.
     - Sessions table schema with UUID7 ID, app name, window title, start/end timestamps, source, and device ID.
     - Heartbeat-merge logic: extend the last session if it matches the current app+title and the gap is ≤ 9s; otherwise, insert a new record.
   - **REST API**:
     - Synchronous `tiny_http` server on `127.0.0.1:5566`.
     - GET `/api/sessions?since=<timestamp>` returning a JSON array of tracked sessions.
     - CORS header: `Access-Control-Allow-Origin: *` to permit the external extension's connections.
   - **Single-Instance Protection**: Win32 named mutex (`Local\ACTLog-Instance-Mutex`).
   - **Clean Shutdown**: `Arc<AtomicBool>` signal shared across threads to exit loops when "Quit" is clicked.

### Out of Scope for v0.0.1
- **Chrome Extension UI**: Deployed and developed in a separate repository.
- **Desktop UI (`actlog-ui.exe` / egui)**: Deferred. The tray app runs headlessly without a separate stats window.
- **Auto-Start Registry/Startup Shortcut**: Defers the installer/shortcut creation using `mslnk`. Run the executable manually for v0.0.1.
- **Auto-Backup (JSON.gz & DB Snapshots)**: Deferred. No scheduled `VACUUM INTO` or compression runs.
- **Crash Recovery Gap Insertion**: Deferred. If the app restarts, it begins tracking immediately without analyzing pre-crash sessions.
- **Import API**: Deferred. Read-only API (`GET` only) is sufficient for the extension client.
- **Config Customization**: Minimal hardcoded values (e.g. 180s idle, 3s poll) or a simple text-based `device_id` storage.

## 2. Directory Layout & File Structure

For the MVP, we will structure the Rust project as a single workspace binary with cleanly separated modules to avoid unnecessary crate-level compilation overhead:

```
actlog/
├── .agents/
│   ├── AGENTS.md
│   ├── features.md
│   └── v0.0.1-mvp.md       (This file)
├── src/                    (Rust Backend Source)
│   ├── main.rs             (Initialization, Mutex, Tray & Threads orchestration)
│   ├── tracker.rs          (Foreground window loop)
│   ├── afk.rs              (Last input info checking loop)
│   ├── merge.rs            (Heartbeat merge logic)
│   ├── db.rs               (SQLite setup, schema creation, read/write helpers)
│   └── server.rs           (tiny_http server thread)
├── Cargo.toml
└── .gitignore
```

## 3. SQLite Database Schema (v0.0.1)

```sql
CREATE TABLE IF NOT EXISTS sessions (
    id TEXT PRIMARY KEY,          -- UUID7 string format
    app TEXT NOT NULL,             -- Executable filename (e.g., "chrome.exe")
    title TEXT NOT NULL,           -- Window title
    start_utc INTEGER NOT NULL,    -- Unix epoch milliseconds
    end_utc INTEGER NOT NULL,      -- Unix epoch milliseconds
    source TEXT NOT NULL,          -- 'foreground' | 'afk'
    device_id TEXT NOT NULL        -- Generated UUID per-machine
);

CREATE INDEX IF NOT EXISTS idx_sessions_start ON sessions(start_utc);
```

## 4. Implementation Steps & Checklist

- [ ] **Phase 1: Project & Database Initialization**
  - Create `Cargo.toml` with dependencies (`windows`, `rusqlite`, `uuid` with v7 feature, `serde`, `serde_json`, `tiny_http`, `tray-icon`, `muda`).
  - Implement path resolution relative to `current_exe()`.
  - Implement `db.rs` to initialize the database file in WAL mode and create the `sessions` table.
  - Implement device ID generation (read/write a simple text file next to exe).

- [ ] **Phase 2: Windows Window Tracker (`core/tracker.rs`)**
  - Implement Win32 window polling via `GetForegroundWindow`.
  - Add UWP window enumeration to unpack `ApplicationFrameHost.exe`.
  - Implement process name resolution and handle `ERROR_ACCESS_DENIED` / NULL window handles.

- [ ] **Phase 3: AFK & Heartbeat Merge**
  - Implement `GetLastInputInfo` wrapper to calculate idle duration.
  - Implement `merge.rs` logic to fetch the latest DB session, check gap limits, and insert or update rows.

- [ ] **Phase 4: REST API & Server**
  - Setup `tiny_http` server on port 5566.
  - Implement `/api/sessions` endpoint retrieving data from the SQLite DB.
  - Handle CORS headers with `Access-Control-Allow-Origin: *` to enable external extension fetches.

- [ ] **Phase 5: Single Instance & Tray App Orchestration**
  - Create the Named Mutex guard in `main.rs`.
  - Setup the tray menu ("Quit") using `tray-icon` and `muda`.
  - Establish the cross-thread shutdown signal using an `Arc<AtomicBool>`.
  - Spawn the Tracker and Server threads and coordinate a clean join sequence.

- [ ] **Phase 6: Verification & Testing**
  - Validate process name query behavior for standard vs elevated apps.
  - Verify database writes and session merging logic.
  - Test CORS compatibility and local REST API data responses using tools like `curl` or browser queries to `http://127.0.0.1:5566/api/sessions`.
